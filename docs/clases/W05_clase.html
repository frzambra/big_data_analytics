¿Qué Vimos la Semana Pasada?

  • El Diagrama de Caso de Uso describe qué hace el sistema y quién interactúa con él.
  • El levantamiento de requerimientos es la base de todo el modelamiento.
  • Actores: Cliente, Repartidor, Administrador.
  • Casos de uso: Realizar pedido, Confirmar entrega, Gestionar producto, etc.

Hoy: ¿Cómo modelamos la estructura interna del sistema? → Diagrama UML de Clases.

¿Qué es el Diagrama UML de Clases?

  • UML = Unified Modeling Language. Estándar internacional para modelar software.
  • El diagrama de clases muestra las entidades del sistema, sus atributos, sus métodos y sus relaciones.
  • Es el paso previo al Modelo Entidad-Relación: muestra la estructura, no solo el comportamiento.

Diferencia con el Caso de Uso:

  • Caso de Uso → ¿Qué hace el sistema?
  • Diagrama de Clases → ¿Cómo está estructurado internamente?

Anatomía de una Clase UML

┌──────────────────────┐
│       Cliente        │  ← Nombre de la clase
├──────────────────────┤
│ - id_cliente: int    │  ← Atributos (con tipo)
│ - nombre: String     │  (- = privado, + = público)
│ - email: String      │
│ - telefono: String   │
│ - region: String     │
├──────────────────────┤
│ + registrarse()      │  ← Métodos (operaciones)
│ + actualizarEmail()  │
│ + getHistorial()     │
└──────────────────────┘

Clases del Sistema TechStyle

Clases identificadas a partir del Caso de Uso:

  • Cliente — el actor que realiza pedidos.
  • Pedido — la transacción central del sistema.
  • Producto — lo que se vende.
  • OrderItem — el ítem individual dentro de un pedido.
  • Repartidor — quien entrega el pedido.
  • Zona — área geográfica de cobertura.
  • Devolución — pedido que el cliente retorna.

Tipos de Relaciones UML

Relación Símbolo Significado
Asociación Línea sólida Dos clases se conocen/usan
Agregación Diamante vacío ◇ “tiene un” (parte puede existir sin el todo)
Composición Diamante relleno ◆ “compuesto de” (parte no existe sin el todo)
Dependencia Flecha punteada Usa temporalmente otra clase
Herencia Flecha vacía ▷ “es un” (subclase de)

Cardinalidades en UML

  • 1 — exactamente uno.
  • 0..1 — cero o uno (opcional).
  • * — cero o muchos.
  • 1..* — uno o muchos.

Ejemplo: Un Cliente puede tener 0..* Pedidos. Un Pedido pertenece a exactamente 1 Cliente.

Del UML al Modelo ER

Correspondencia directa:

Elemento UML Elemento ER
Clase Entidad
Atributo de clase Atributo de entidad
Atributo con id Identificador / PK
Asociación 1:N Relación con FK en el lado N
Asociación N:M con clase propia Entidad asociativa
