¿Qué Vimos la Semana Pasada?

  • La normalización elimina redundancias y anomalías de las tablas.
  • 1FN: valores atómicos, sin grupos repetitivos.
  • 2FN: todos los atributos dependen de la PK completa.
  • 3FN: no hay dependencias transitivas entre atributos no-clave.

Hoy: ¿Cómo documentamos formalmente el modelo relacional? Y ¿qué herramienta usaremos para construir la base de datos?

El Problema de la Documentación

Situación en TechStyle:

El analista Juan diseñó el modelo relacional. Ahora está de vacaciones. Un nuevo desarrollador debe implementar la base de datos.

  • ¿Qué tipo de dato tiene la columna total en pedidos? ¿INT o DECIMAL?
  • ¿Puede ser NULL la columna telefono en clientes?
  • ¿Cuál es el valor por defecto del campo estado en pedidos?
  • ¿Qué significa un estado = 'cancelado'? ¿Es diferente a 'rechazado'?

Sin documentación formal → errores de implementación → datos incorrectos → decisiones erróneas.

Aplicación: El Costo de No Documentar

Caso real en TechStyle:

  • El desarrollador implementó total como INT (sin decimales).
  • TechStyle cobra $12.990 por algunas poleras.
  • El sistema guardaba $12.990 como $12 (truncado) → pérdida de centavos.
  • En 8 meses, el CMI mostraba ingresos 0,1% menores a los reales.
  • Roberto tomó decisiones de inversión con datos incorrectos.

Un tipo de dato mal elegido por falta de documentación → decisiones incorrectas en la cima de la organización.

¿Qué es el Diccionario de Datos?

  • Documento formal que describe cada tabla y cada atributo del modelo relacional.
  • Es el “manual técnico” de la base de datos.
  • Permite a cualquier desarrollador o analista entender la estructura sin preguntar al diseñador original.

Contenido mínimo por atributo: Nombre · Tipo de dato · Longitud · Permite NULL · Valor por defecto · Descripción · Restricciones

Explicar por qué DECIMAL y no FLOAT para precios: DECIMAL es exacto, FLOAT tiene errores de redondeo que son inaceptables en dinero.

Aplicación: Elegir el Tipo de Dato Correcto

¿Por qué DECIMAL y no FLOAT para precios?

  • FLOAT usa representación binaria → puede almacenar $12.990 como $12.989999999998.
  • DECIMAL(10,2) almacena exactamente $12.99 → sin errores de redondeo.
  • En un sistema con millones de transacciones, los errores de FLOAT se acumulan.

Regla: Todo valor monetario → DECIMAL. Todo ID o contador → INT. Todo texto de longitud variable → VARCHAR. Todo texto libre sin límite → TEXT.

Verificación: ¿Qué Tipo de Dato Usarías?

Elige el tipo de dato más apropiado para cada atributo:

  1. Número de RUT del cliente (ej: “12345678-9”) → VARCHAR(12) (tiene guión, no es número)
  2. Precio de un producto (ej: 45.000,00) → DECIMAL(10,2)
  3. Si un repartidor está activo o de baja → BOOLEAN (o TINYINT(1))
  4. Descripción larga de un producto (párrafo) → TEXT
  5. Mes de registro de un cliente (solo el mes, 1-12) → TINYINT (entero pequeño)

¿Por qué MySQL?

  • Comparación con Excel: Excel maneja filas, MySQL maneja millones de filas eficientemente.
  • Comparación con Power BI: Power BI consume datos, MySQL los almacena y gestiona.
  • Por qué MySQL específicamente:
    • Open source y gratuito.
    • Estándar en la industria (e-commerce, retail, startups).
    • Compatible con Power BI, Python, y la mayoría de herramientas de análisis.

MySQL Workbench = interfaz gráfica para escribir y ejecutar consultas SQL. El servidor MySQL es el motor que procesa las consultas y gestiona los archivos de datos.

Primera Conexión a MySQL Workbench

Pasos para conectarse:

  1. Abrir MySQL Workbench.
  2. Hacer clic en + en “MySQL Connections”.
  3. Configurar: hostname 127.0.0.1, puerto 3306, usuario root.
  4. Ingresar la contraseña definida durante la instalación.
  5. Hacer clic en “Test Connection” → debe aparecer “Successfully made the MySQL connection”.

En el laboratorio: Todos instalarán y configurarán MySQL Workbench hoy.

Discusión: ¿Por qué los Tipos de Datos son Decisiones de Negocio?

Caso: El campo total en pedidos. ¿INT, FLOAT o DECIMAL?

  • INT: No permite centavos → imposible para precios reales.
  • FLOAT: Tiene errores de redondeo → $45.000,00 puede guardarse como $44.999,9999.
  • DECIMAL(10,2): Exacto hasta 2 decimales → correcto para valores monetarios.

Conclusión: Elegir el tipo de dato incorrecto genera errores contables reales. Un analista que no entiende tipos de datos puede comprometer la integridad financiera de la organización.

Conexión con la Toma de Decisiones

  • El diccionario de datos garantiza que todos los sistemas que consumen la base de datos (Power BI, reportes, APIs) interpretan los datos de la misma forma.
  • Un atributo mal tipificado puede hacer que una consulta del CMI devuelva resultados incorrectos.
  • La documentación es parte integral del diseño, no un trámite opcional.

Puntos Clave

  • El diccionario de datos es el documento formal que describe cada tabla y atributo del modelo relacional.
  • Cada atributo debe tener: tipo de dato, longitud, NULL permitido, valor por defecto, descripción y restricciones.
  • MySQL es el motor de base de datos relacional que usaremos: open source, estándar industrial.
  • MySQL Workbench es la interfaz gráfica para escribir SQL y administrar la base de datos.
  • Los tipos de datos son decisiones de negocio: elegir FLOAT en vez de DECIMAL para precios genera errores financieros reales.
