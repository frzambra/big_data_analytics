Aplicación: DDL como Traducción del Diccionario

El DDL traduce el diccionario de datos a código ejecutable:

Diccionario de datos DDL en MySQL
Tabla clientes CREATE TABLE clientes (...)
id_cliente: INT, No NULL, AUTO id_cliente INT NOT NULL AUTO_INCREMENT
email: VARCHAR(150), No NULL, UNIQUE email VARCHAR(150) NOT NULL, UNIQUE KEY uq_email (email)
PK: id_cliente PRIMARY KEY (id_cliente)
FK: id_cliente → clientes FOREIGN KEY (id_cliente) REFERENCES clientes(id_cliente)

Crear la Base de Datos

-- Crear la base de datos
CREATE DATABASE techstyle_db
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_spanish_ci;

-- Seleccionarla como activa
USE techstyle_db;
  • utf8mb4: soporte completo para caracteres españoles (tildes, ñ) y emojis.
  • COLLATE spanish_ci: ordenamiento insensible a mayúsculas en español (a = A, á ≠ a).

Verificación: Antes de Crear Tablas

Orden correcto de creación (respetando dependencias de FK):

  1. Primero las tablas sin FK (sin dependencias): zonas, clientes, productos
  2. Luego las tablas que dependen de las anteriores: repartidores (FK → zonas)
  3. Después las tablas con más dependencias: pedidos (FK → clientes + repartidores)
  4. Finalmente las tablas hoja: order_items (FK → pedidos + productos), devoluciones (FK → pedidos)

Si creas tablas en el orden incorrecto → error de FK al intentar referenciar una tabla que no existe aún.

Crear la Primera Tabla: zonas

CREATE TABLE zonas (
    id_zona    INT           NOT NULL AUTO_INCREMENT,
    nombre     VARCHAR(50)   NOT NULL,
    region     VARCHAR(50)   NOT NULL,
    PRIMARY KEY (id_zona)
);

Empezar por zonas porque no tiene dependencias de FK. Seguir el orden correcto: primero las tablas sin FK, luego las que sí las tienen.

Crear la Tabla clientes

CREATE TABLE clientes (
    id_cliente      INT           NOT NULL AUTO_INCREMENT,
    nombre          VARCHAR(100)  NOT NULL,
    email           VARCHAR(150)  NOT NULL,
    telefono        VARCHAR(20)   NULL,
    fecha_registro  DATE          NOT NULL DEFAULT (CURDATE()),
    region          VARCHAR(50)   NOT NULL,
    PRIMARY KEY (id_cliente),
    UNIQUE KEY uq_email (email)
);
  • AUTO_INCREMENT: MySQL genera automáticamente el ID.
  • UNIQUE KEY: garantiza que no haya dos clientes con el mismo email.
  • DEFAULT (CURDATE()): si no se especifica fecha, usa la fecha actual.

Crear la Tabla repartidores

CREATE TABLE repartidores (
    id_repartidor  INT          NOT NULL AUTO_INCREMENT,
    nombre         VARCHAR(100) NOT NULL,
    telefono       VARCHAR(20)  NULL,
    id_zona        INT          NOT NULL,
    PRIMARY KEY (id_repartidor),
    CONSTRAINT fk_rep_zona
        FOREIGN KEY (id_zona) REFERENCES zonas(id_zona)
);
  • CONSTRAINT fk_rep_zona: nombre explícito para la restricción de FK (facilita el diagnóstico de errores).
  • La FK id_zona debe existir en zonas antes de insertar un repartidor.

Crear las Tablas Centrales

CREATE TABLE productos (
    id_producto  INT            NOT NULL AUTO_INCREMENT,
    nombre       VARCHAR(150)   NOT NULL,
    categoria    VARCHAR(50)    NOT NULL,
    precio       DECIMAL(10,2)  NOT NULL,
    stock        INT            NOT NULL DEFAULT 0,
    PRIMARY KEY (id_producto)
);

CREATE TABLE pedidos (
    id_pedido      INT            NOT NULL AUTO_INCREMENT,
    fecha_pedido   DATE           NOT NULL DEFAULT (CURDATE()),
    total          DECIMAL(10,2)  NOT NULL,
    estado         VARCHAR(30)    NOT NULL DEFAULT 'pendiente',
    id_cliente     INT            NOT NULL,
    id_repartidor  INT            NULL,
    PRIMARY KEY (id_pedido),
    CONSTRAINT fk_ped_cli  FOREIGN KEY (id_cliente)    REFERENCES clientes(id_cliente),
    CONSTRAINT fk_ped_rep  FOREIGN KEY (id_repartidor) REFERENCES repartidores(id_repartidor)
);

Crear la Tabla Intermedia y Devoluciones

CREATE TABLE order_items (
    id_item         INT            NOT NULL AUTO_INCREMENT,
    id_pedido       INT            NOT NULL,
    id_producto     INT            NOT NULL,
    cantidad        INT            NOT NULL,
    precio_unitario DECIMAL(10,2)  NOT NULL,
    PRIMARY KEY (id_item),
    CONSTRAINT fk_item_ped  FOREIGN KEY (id_pedido)   REFERENCES pedidos(id_pedido),
    CONSTRAINT fk_item_prod FOREIGN KEY (id_producto) REFERENCES productos(id_producto)
);

CREATE TABLE devoluciones (
    id_devolucion  INT            NOT NULL AUTO_INCREMENT,
    fecha          DATE           NOT NULL,
    motivo         VARCHAR(200)   NOT NULL,
    monto          DECIMAL(10,2)  NOT NULL,
    id_pedido      INT            NOT NULL,
    PRIMARY KEY (id_devolucion),
    CONSTRAINT fk_dev_ped FOREIGN KEY (id_pedido) REFERENCES pedidos(id_pedido)
);

Verificar la Estructura Creada

-- Ver todas las tablas de techstyle_db

-- Ver la estructura de una tabla específica

-- Ver el DDL completo de una tabla
SHOW CREATE TABLE pedidos\G

Orden correcto de creación (respetando dependencias de FK): 1. zonas → 2. clientes → 3. repartidores → 4. productos → 5. pedidos → 6. order_items → 7. devoluciones

Modificar Tablas: ALTER TABLE

-- Agregar una columna nueva
ALTER TABLE clientes ADD COLUMN activo BOOLEAN NOT NULL DEFAULT TRUE;

-- Modificar el tipo de una columna existente
ALTER TABLE productos MODIFY COLUMN categoria VARCHAR(80) NOT NULL;

-- Agregar una restricción FK después de crear la tabla
ALTER TABLE pedidos
ADD CONSTRAINT fk_ped_zona
FOREIGN KEY (id_zona) REFERENCES zonas(id_zona);

-- Eliminar una columna
ALTER TABLE clientes DROP COLUMN activo;

Advertencia: DROP COLUMN elimina la columna y todos sus datos permanentemente.

Aplicación: ¿Qué pasa si violo una restricción de FK?

-- Intentar insertar un pedido con un cliente que no existe
INSERT INTO pedidos (fecha_pedido, total, estado, id_cliente)
VALUES ('2026-06-01', 55000, 'pendiente', 9999);
-- ERROR 1452: Cannot add or update a child row:
-- a foreign key constraint fails (`techstyle_db`.`pedidos`,
-- CONSTRAINT `fk_ped_cli` FOREIGN KEY (`id_cliente`)
-- REFERENCES `clientes` (`id_cliente`))

MySQL protege la integridad referencial automáticamente.

→ Esto garantiza que no existan pedidos de “clientes fantasma” en el CMI.

Verificación: ¿Qué Restricción Corresponde?

Elige la restricción SQL correcta para cada requerimiento de negocio:

  1. “El email de un cliente debe ser único en el sistema” → UNIQUE KEY
  2. “El precio de un producto no puede ser negativo” → CHECK (precio >= 0)
  3. “Si no se especifica estado del pedido, debe ser ‘pendiente’” → DEFAULT ‘pendiente’
  4. “Todo pedido debe tener un cliente asociado” → FOREIGN KEY NOT NULL
  5. “El id_pedido debe incrementarse automáticamente” → AUTO_INCREMENT

Conexión con la Toma de Decisiones

  • Las restricciones NOT NULL, UNIQUE y FOREIGN KEY son la implementación técnica de las reglas de negocio.
  • Si un analista dice “cada pedido debe tener un cliente”, eso se traduce en FOREIGN KEY NOT NULL.
  • La base de datos hace cumplir las reglas del negocio de forma automática.

“Sin restricciones de integridad, la base de datos es solo un archivo de texto glorificado. Las restricciones son lo que hace que la base de datos proteja el negocio.”

Puntos Clave

  • DDL (Data Definition Language) define la estructura de la base de datos: CREATE, ALTER, DROP.
  • El orden de creación de tablas importa: primero las tablas sin FK, luego las que dependen.
  • Las restricciones traducen reglas de negocio a código: PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE, DEFAULT.
  • CONSTRAINT nombre_fk permite identificar la restricción de FK con un nombre descriptivo.
  • DECIMAL(10,2) para precios: exacto. INT para IDs. VARCHAR(n) para texto variable.
