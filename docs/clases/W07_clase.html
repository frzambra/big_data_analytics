¿Qué Vimos la Semana Pasada?

  • El Modelo ER representa la realidad del negocio como entidades, atributos y relaciones.
  • La cardinalidad (1:1, 1:N, N:M) define las reglas de negocio.
  • Las relaciones N:M generan una entidad asociativa (tabla intermedia).

Hoy: ¿Cómo convertimos el diagrama ER en tablas de una base de datos real?

Del Diagrama al Sistema Real

El modelo ER es el plano. El modelo relacional es la construcción.

  • El modelo relacional es la base teórica de bases de datos como MySQL, PostgreSQL y SQL Server.
  • Organiza los datos en tablas (relaciones) con filas y columnas.
  • Las llaves garantizan la integridad de los datos.

Inventado por Edgar Codd (IBM, 1970) — fundamento de toda base de datos relacional moderna.

Conceptos del Modelo Relacional

Concepto Descripción
Tabla (Relación) Conjunto de filas con el mismo esquema
Fila (Tupla) Un registro de datos
Columna (Atributo) Una característica del registro
Dominio Conjunto de valores válidos para un atributo
Llave Primaria (PK) Identifica de forma única cada fila
Llave Foránea (FK) Referencia a la PK de otra tabla

Reglas de Transformación ER → Relacional

Regla 1: Entidad → Tabla - Cada entidad del ER se convierte en una tabla. - Los atributos de la entidad se convierten en columnas. - El identificador único se convierte en la PK.

Regla 2: Relación 1:N → Llave Foránea - La FK se agrega en la tabla del lado N. - Ejemplo: id_cliente (FK) en la tabla Pedidos.

Regla 3: Relación N:M → Tabla Intermedia - Se crea una tabla nueva con las PK de ambas entidades como FK. - Puede agregar atributos propios de la relación.

Transformación TechStyle: Paso a Paso

Entidad Cliente:

clientes(id_cliente PK, nombre, email, telefono, fecha_registro, region)

Entidad Pedido (con FK a Cliente):

pedidos(id_pedido PK, fecha_pedido, total, estado, id_cliente FK→clientes)

Entidad Producto:

productos(id_producto PK, nombre, categoria, precio, stock)

Tabla Intermedia OrderItem (N:M entre Pedido y Producto):

order_items(id_item PK, id_pedido FK→pedidos, id_producto FK→productos, cantidad, precio_unitario)

Continuación: Repartidor, Zona y Devolución

Entidad Zona:

zonas(id_zona PK, nombre_zona, region)

Entidad Repartidor (con FK a Zona):

repartidores(id_repartidor PK, nombre, telefono, id_zona FK→zonas)

Tabla Pedidos actualizada (con FK a Repartidor):

pedidos(id_pedido PK, fecha_pedido, total, estado,
        id_cliente FK→clientes, id_repartidor FK→repartidores)

Entidad Devolución (opcional, FK a Pedido):

devoluciones(id_devolucion PK, fecha, motivo, monto_devuelto, id_pedido FK→pedidos)

Integridad Referencial

Regla fundamental: Una FK debe apuntar a una PK que existe.

  • No puedo registrar un pedido con id_cliente = 999 si el cliente 999 no existe.
  • No puedo eliminar un cliente que tiene pedidos asociados (sin antes eliminar o reasignar esos pedidos).
  • Esto se llama restricción de integridad referencial y la base de datos la hace cumplir automáticamente.

¿Por qué importa?: Sin integridad referencial, el CMI puede mostrar ventas de “clientes fantasma”.

Ética en el Diseño Relacional

Caso: TechStyle almacena el número de RUT de cada cliente para facturación.

  • ¿Qué nivel de acceso debe tener cada actor del sistema a esta columna?
  • ¿Deben almacenarse datos sensibles en texto plano o cifrados?
  • Principio de mínimo privilegio: Cada actor accede solo a los datos que necesita para su función.

Responsabilidad del diseñador: El modelo relacional define qué datos se guardan y cómo se protegen.
