Situación: TechStyle en el Año 2028

TechStyle creció exponencialmente:

  • De 450.000 a 8.500.000 clientes registrados
  • De 2.3M a 47 millones de órdenes históricas
  • Expandida a 6 países de Latinoamérica
  • Registra clicks, tiempo en página, búsquedas por cada usuario
  • Integra datos de redes sociales, clima y eventos locales

El sistema de base de datos actual colapsa cada fin de semana.

TechStyle ya no tiene un problema de “sistemas de información”. Tiene un problema de Big Data.

El salto conceptual es importante: el mismo tipo de problema (capturar, almacenar, procesar datos) pero a una escala que requiere arquitecturas completamente diferentes.

¿Qué es Big Data?

No es solo “muchos datos”.

Big Data se define por un conjunto de características conocidas como las 5 V:

  Volumen · Velocidad · Variedad · Veracidad · Valor

Cada “V” representa un desafío distinto que los sistemas tradicionales no pueden resolver.

Originalmente eran 3 V (Laney, 2001). Luego se agregaron Veracidad y Valor. El punto no es memorizar las V, sino entender que Big Data es cualitativamente diferente a los datos tradicionales.

V1: Volumen · V2: Velocidad

Volumen: la cantidad es masiva

  • TechStyle 2028: miles de millones de registros
  • Se mide en Terabytes y Petabytes
  • Los sistemas relacionales no escalan bien

Velocidad: se genera y debe procesarse rápido

  • 12.000 transacciones/minuto en Black Friday
  • Stock debe actualizarse en milisegundos
  • Detección de fraude: menos de 100ms

TechStyle ya no puede esperar un reporte diario — necesita información en tiempo real.

Volumen solo no hace Big Data. Velocidad tampoco. Es la combinación de las 5 V lo que crea el desafío. Un banco tiene ambas características: millones de transacciones diarias que deben procesarse instantáneamente.

V3: Variedad · V4: Veracidad · V5: Valor

V Descripción Ejemplo TechStyle
Variedad Datos de múltiples formatos y fuentes Órdenes (tablas) + Reseñas (texto) + Fotos + Clicks (logs)
Veracidad No todos los datos son confiables Reseñas falsas entre 500.000 mensuales
Valor Los datos solo valen si generan decisiones útiles ¿Vale almacenar cada movimiento del cursor?

La V más importante es el Valor: si no sabes para qué sirve el dato, no vale la pena recolectarlo.

La Variedad es el cambio más dramático. Las BD relacionales manejan bien datos estructurados (tablas), pero son rígidas con texto, imágenes o logs. El Valor conecta Big Data con el marco central del curso: los datos sirven para tomar decisiones.

Del Sistema Operacional al Analítico

TechStyle tiene 2 necesidades de datos radicalmente distintas:

  NECESIDAD 1: Registrar cada venta en tiempo real
  → Rápido, muchas escrituras simultáneas, datos actuales

  NECESIDAD 2: Analizar 5 años de ventas para decidir
  → Complejo, pocas escrituras, grandes lecturas históricas

¿Puede el mismo sistema hacer ambas eficientemente?

No. Por eso existen 2 arquitecturas distintas: OLTP y OLAP.

Esta tensión es la motivación central para OLTP/OLAP. No es una preferencia técnica: son requerimientos que se contradicen entre sí.

OLTP vs. OLAP

Característica OLTP OLAP
Propósito Registrar operaciones Analizar histórico
Usuarios Miles (clientes, operarios) Pocos (analistas, gerentes)
Operaciones Muchas escrituras pequeñas Pocas lecturas complejas
Datos Actuales Históricos (años)
Respuesta Milisegundos Segundos a minutos
Ejemplo Sistema de ventas online Dashboard de BI / CMI

TechStyle “¿está disponible este producto?” → OLTP

“¿cuál fue la categoría más vendida en los últimos 3 años?” → OLAP

Preguntas sobre “ahora” van a OLTP; preguntas sobre “patrones históricos” van a OLAP. Esta distinción es conceptualmente importante y prácticamente útil en cualquier trabajo de datos.

El Data Warehouse: Integrando Todo

Los datos van del OLTP al OLAP mediante transformación

┌─────────────────────────────────────┐
│         FUENTES OLTP                │
│  BD Ventas · BD Clientes · CRM      │
└──────────────────┬──────────────────┘
                   │  ETL (cada noche)
                   ▼
        ┌──────────────────────┐
        │    DATA WAREHOUSE    │
        │   (OLAP · Historial) │
        └──────────┬───────────┘
                   │
      ┌────────────┼────────────┐
      ▼            ▼            ▼
   [BI / CMI]  [Reportes]  [Analítica]

El Data Warehouse resuelve el problema de silos: integra datos de múltiples sistemas OLTP en un repositorio coherente optimizado para análisis. Es el corazón de la arquitectura de Business Intelligence.

ETL: El Proceso de Integración

Extract · Transform · Load

  • Extraer: obtener datos de los sistemas OLTP fuente
  • Transformar: limpiar, estandarizar, integrar, aplicar calidad de datos
  • Cargar: insertar en el Data Warehouse

TechStyle: cada noche a las 2am, el proceso ETL extrae datos de ventas, bodega y CRM; los limpia y estandariza; y los carga en el Data Warehouse.

A las 7am el analista tiene datos frescos para el CMI del día.

ETL es donde se “paga el precio” de la integración. La transformación puede ser compleja: sistemas con formatos distintos, monedas distintas, nombres distintos para el mismo concepto. Aquí es donde la calidad de datos (clase anterior) se convierte en proceso sistemático.

El Modelo Completo de TechStyle

   [Clientes compran]  →  OLTP  →  datos en tiempo real
                                         ↓
                                   ETL (noche)
                                         ↓
                               DATA WAREHOUSE (OLAP)

Síntesis: Los 5 Conceptos Clave de la Unidad 1

  1. Sistema de Información — conjunto de componentes que transforma datos en información para apoyar decisiones. Tiene 5 elementos: hardware, software, datos, personas, procesos.

  2. Niveles y Tipos de SI — 3 niveles (operativo, táctico, estratégico) con SI distintos (TPS, BI, DSS, CRM, EIS).

  3. CMI — 4 perspectivas (Financiera, Cliente, Procesos, Aprendizaje) con KPIs que conectan estrategia y datos.

  4. Calidad de Datos — 4 dimensiones (exactitud, completitud, consistencia, oportunidad). Datos malos = decisiones malas.

  5. Big Data y Arquitectura — 5 V, OLTP (operacional), OLAP (analítico), Data Warehouse, ETL.

Este repaso es el corazón de la preparación para la solemne. Pedir a los estudiantes que anoten qué concepto sienten menos claro para abordarlo en la actividad.

Caso Integrador: ServiRápido

ServiRápido: empresa de mantención del hogar (gasfitería, electricidad, cerrajería) con 180 técnicos y 3.200 servicios mensuales en Santiago.

Actualmente gestiona sus datos con: - Google Sheets para registrar servicios - WhatsApp para coordinar con técnicos - Email para enviar facturas - Notas en papel para inventario de repuestos

Problema: No saben si son rentables por zona ni qué técnicos tienen mejor desempeño.

Su rol: analista de datos contratado para diseñar el SI de ServiRápido.

Este escenario refleja la realidad de miles de PYMEs en Chile. Tiene todos los problemas vistos en la unidad: datos dispersos, mala calidad, sin arquitectura analítica.

Actividad: Diseñar el SI de ServiRápido

En grupos de 4-5 personas (15 minutos):

Parte 1 — Diagnóstico: identifiquen 2 problemas de calidad de datos y 2 barreras al flujo de información.

Parte 2 — Diseño del SI: propongan qué tipos de SI necesita ServiRápido y por qué (TPS, BI, CRM, etc.).

Parte 3 — CMI: diseñen 1 objetivo y 1 KPI por cada perspectiva del CMI.

Presentación: 2 minutos por grupo.

Parte 1: datos en papel → exactitud y completitud en riesgo; WhatsApp → no estructurado, no analizable. Parte 2: TPS para registro de servicios, CRM para clientes, BI para análisis de rentabilidad por zona. Parte 3: Financiero=margen por servicio, Cliente=tasa de recontratación, Procesos=tiempo de respuesta, Aprendizaje=capacitación de técnicos.

Lo Que Aprendimos Hoy

  1. Big Data se define por las 5 V: Volumen, Velocidad, Variedad, Veracidad, Valor
  2. Los sistemas OLTP registran operaciones en tiempo real; los OLAP analizan histórico
  3. El Data Warehouse integra datos de múltiples fuentes para análisis
  4. El proceso ETL conecta los sistemas operacionales con el almacén analítico
  5. Todos los conceptos de la Unidad 1 forman una cadena completa: dato → decisión

Big Data no es el objetivo: el objetivo son las decisiones que habilita.
