Hoy iniciamos la Unidad 2: ¿Cómo se modela un sistema de información antes de construir la base de datos?

Resumen de la Unidad 1 en una Frase

“Los Sistemas de Información convierten datos crudos en decisiones estratégicas — pero solo funcionan si el modelo de datos que los alimenta refleja fielmente la realidad del negocio.”

La Unidad 2 se dedica exactamente a eso: modelar la realidad del negocio antes de construir la base de datos.

Secuencia: Caso de Uso → UML → Modelo ER → Modelo Relacional → (Unidad 3) Implementación en MySQL.

El Problema de TechStyle: Crecen y No Saben Qué Construir

Situación real:

TechStyle decide construir un nuevo módulo de pedidos. El equipo de TI necesita entender qué debe hacer el sistema antes de escribir una sola línea de código o crear una tabla.

Pregunta disparadora: ¿Cómo le explicamos a un desarrollador qué debe hacer el sistema, usando el lenguaje del negocio?

Conectar con la experiencia de los estudiantes usando Power BI: ellos ya modelaron datos, ahora aprenderán a modelar sistemas.

¿Qué es el Levantamiento de Requerimientos?

  • Proceso de identificar qué necesita hacer el sistema.
  • Se realiza antes del diseño técnico.
  • Fuentes: entrevistas con usuarios, observación de procesos, documentos actuales.

Salida del levantamiento: una lista de requerimientos funcionales y no funcionales.

Requerimiento funcional: “El sistema debe permitir que el cliente realice un pedido.”

Requerimiento no funcional: “El sistema debe responder en menos de 2 segundos.”

Técnicas de Levantamiento de Requerimientos

Cómo obtiene Juan la información que necesita del negocio:

  • Entrevistas individuales: conversar con Sofía, María y Roberto para entender qué necesita cada actor.
  • Observación directa: acompañar a Sofía en su ruta para entender cómo usa el sistema hoy.
  • Talleres facilitados: reunir a todos los actores y modelar el proceso juntos (story mapping).
  • Revisión de documentos: analizar los formularios en papel, hojas Excel y correos actuales.
  • Prototipado: mostrar una versión preliminar del sistema y recoger feedback temprano.

Aplicación: Entrevista con María González

Juan entrevista a María para levantar requerimientos:

  • Juan: “María, ¿qué necesitas saber sobre los pedidos para tomar decisiones de campaña?”
  • María: “Necesito saber qué categorías venden más, en qué zonas y en qué fechas.”
  • Juan: “¿Con qué frecuencia necesitas esta información?”
  • María: “Cada semana, los lunes a primera hora.”
  • Juan: “¿Qué pasa si la información llega el martes?”
  • María: “Ya es tarde — la reunión de presupuesto es el lunes.”

Requerimiento extraído: El sistema debe generar el reporte de ventas por categoría y zona todos los domingos antes de las 23:59. (Funcional + No funcional de oportunidad)

Verificación: ¿Funcional o No Funcional?

Clasifica cada requerimiento:

  1. “El sistema debe permitir al repartidor marcar un pedido como entregado desde su app” → Funcional
  2. “El sistema debe soportar 10.000 usuarios simultáneos sin degradar el tiempo de respuesta” → No funcional (rendimiento)
  3. “El sistema debe enviar una notificación al cliente cuando el pedido está en camino” → Funcional
  4. “El sistema debe estar disponible el 99,9% del tiempo” → No funcional (disponibilidad)
  5. “El sistema debe permitir al Administrador desactivar un producto del catálogo” → Funcional

El rectángulo delimita claramente qué está dentro del sistema y qué es externo. Nada fuera del rectángulo es responsabilidad del sistema.

Aplicación: include vs extend en TechStyle

Dos relaciones que se confunden frecuentemente:

  • <<include>>: “Realizar pedido” siempre incluye “Procesar pago” — no existe un pedido sin pago. Es obligatorio, no opcional.
  • <<extend>>: “Realizar pedido” puede extenderse con “Solicitar devolución” — pero no en todos los casos. Solo cuando el pedido fue entregado y el cliente no quedó satisfecho.

Regla mnemotécnica: - include = “siempre pasa” → como el IVA en una boleta - extend = “a veces pasa” → como un seguro de envío opcional

Verificación: ¿Include o Extend?

Clasifica la relación entre cada par de casos de uso:

  1. “Registrar pedido” → “Validar stock” → <<include>> (siempre se valida el stock)
  2. “Confirmar entrega” → “Enviar encuesta de satisfacción” → <<extend>> (opcional: solo si el cliente aceptó recibirlas)
  3. “Crear cuenta” → “Verificar email” → <<include>> (siempre se verifica el email)
  4. “Consultar historial” → “Exportar a PDF” → <<extend>> (el cliente puede o no querer exportar)

Actores del Módulo de Pedidos TechStyle

  • Cliente: Navega el catálogo, realiza pedidos, paga, consulta estado, solicita devoluciones.
  • Repartidor: Ve pedidos asignados, confirma entrega.
  • Administrador: Gestiona catálogo de productos, procesa devoluciones, ve reportes.

Regla de oro: Un actor es externo al sistema. No es un módulo del sistema, es quien usa el sistema.

Reglas para Identificar Actores

Un buen actor cumple estas condiciones:

  • Es externo al sistema (no es un componente de software).
  • Tiene un rol definido en relación al sistema (no es solo “una persona”).
  • Inicia o responde a casos de uso.
  • No se implementa — solo interactúa.

Ejemplo de error común: “Sistema de pago externo (Transbank)” puede ser un actor (actor externo de sistema) si el alcance del diagrama lo incluye. “Base de datos” no es un actor — es parte del sistema.

¿Es un Actor o un Módulo del Sistema?

Clasifica cada elemento como Actor o como parte interna del sistema:

  1. “Módulo de facturación” → Parte del sistema (es software interno)
  2. “Gerente de logística que aprueba rutas” → Actor (externo, tiene rol)
  3. “Scheduler que ejecuta el ETL a las 2am” → Actor externo de sistema (si se modela a ese nivel)
  4. “Base de datos MySQL” → Parte del sistema (es infraestructura interna)
  5. “Servicio de SMS (Twilio)” → Actor externo de sistema (sistema externo que interactúa)

Casos de Uso Principales

  • Realizar pedido ← Cliente
  • Seleccionar producto ← Cliente
  • Procesar pago ← Cliente (siempre junto a Realizar pedido → <<include>>)
  • Consultar estado de pedido ← Cliente
  • Solicitar devolución ← Cliente (solo si pedido entregado → <<extend>>)
  • Confirmar entrega ← Repartidor
  • Gestionar producto ← Administrador
  • Procesar devolución ← Administrador

Errores Comunes en Diagramas de Caso de Uso

Los 5 errores más frecuentes (y cómo evitarlos):

  1. Poner procesos como casos de uso: “Validar contraseña” no es un CU — es parte del flujo interno de “Iniciar sesión”.
  2. Olvidar el límite del sistema: sin el rectángulo, no queda claro qué está dentro y qué fuera.
  3. Confundir include/extend: usar <<extend>> donde siempre ocurre (debería ser <<include>>).
  4. Demasiados actores: “Gerente de ventas” y “Subgerente de ventas” que hacen exactamente lo mismo → un solo actor “Administrador”.
  5. Casos de uso demasiado detallados: “Hacer clic en el botón Comprar” no es un caso de uso — es un paso de interfaz.

¿Por Qué Modelar Antes de Construir?

Caso TechStyle:

Construyeron el primer módulo de pedidos sin modelamiento. Después de 6 meses, descubrieron que olvidaron registrar la zona de entrega de cada pedido.

  • No pueden calcular el KPI “tiempo de entrega por zona” del CMI.
  • Deben rediseñar la base de datos, migrar datos y rehacer reportes.
  • Costo estimado: 3 meses de trabajo adicional.

Conclusión: Un diagrama de 2 horas evita 3 meses de retrabajo.

Impacto en el CMI: Modelamiento y Decisiones

  • Si “consultar estado de pedido” no está en el Caso de Uso → no se diseña → no hay pantalla para el cliente → el cliente llama al call center → costo operativo innecesario.
  • Si “solicitar devolución” no está en el CU → no hay datos de devolución → el CMI no puede calcular la “tasa de devolución por zona”.
  • Si “confirmar entrega” no está en el CU → no hay registro de fecha real de entrega → el KPI “tiempo de entrega” es incalculable.

Cada caso de uso omitido es un KPI que no podrá calcularse.

Conexión con la Toma de Decisiones

  • Un Caso de Uso bien hecho garantiza que ningún proceso crítico del negocio quede fuera del sistema.
  • Si “solicitar devolución” no está en el Caso de Uso → no se diseña → no hay datos → el CMI no puede calcular la tasa de devolución.
  • Sin buen modelamiento → sin buenos datos → sin buenas decisiones.

Puntos Clave

  • El levantamiento de requerimientos identifica qué debe hacer el sistema antes de cualquier decisión técnica.
  • Los requerimientos son funcionales (qué hace) o no funcionales (cómo lo hace: rendimiento, disponibilidad).
  • El Diagrama de Caso de Uso muestra actores, acciones del sistema y sus relaciones en lenguaje del negocio.
  • <<include>>: el caso de uso base siempre ejecuta el incluido. <<extend>>: solo en condiciones específicas.
  • Cada caso de uso omitido en el diagrama es un proceso que no se modelará, no se implementará y no generará datos.
